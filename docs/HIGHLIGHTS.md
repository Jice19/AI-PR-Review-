# 项目亮点展示 — 讲解稿大纲

> 录制视频时按此大纲讲解，每个亮点点到为止，重点展示运行效果。预计时长 8-12 分钟。

---

## 一、问题定义 (Why)

**痛点**：PR Review 耗时大、人工审查覆盖面有限、不同 reviewer 关注点不一致。

**解决思路**：用 AI 自动做第一轮 Review，给人工 reviewer 提供决策依据，而非替代人工判断。

---

## 二、核心亮点概览

| # | 亮点 | 一句话总结 |
|---|------|-----------|
| 1 | 全流程自动化 | 输入 PR URL → 自动获取代码 → AI 分析 → 生成报告 + 回贴评论 |
| 2 | 双模型分层策略 | Flash 做大量文件扫描（快+省），Pro 做深度分析（准+深） |
| 3 | RAG 反馈闭环 | 用户标记误报/有用 → 向量化存储 → 下次分析参考历史案例 → 越用越准 |
| 4 | GitHub Actions 免服务 | 不用部署服务器，PR 触发 CI 自动跑分析，结果回贴评论 |
| 5 | SSE 流式体验 | 分析过程实时推送到前端，逐文件浮现问题，不用等全部完成 |
| 6 | 多维度评分 & 决策 | 综合严重程度 + 置信度 + PR 规模，输出明确的 APPROVE/COMMENT/CHANGE |

**操作**：展示反馈标记流程

**讲解**：
- "每个问题用户可以标记为 '有用' 或 '误报'"

- "下次分析类似代码时，系统自动检索历史案例注入 prompt"
- "LLM 参考历史正确案例的模式，同时避免已知的误报模式"
- "这样就形成了反馈闭环，分析准确率会随使用次数不断提升"

## 五、技术决策 (展示技术深度)

录制时可选择 2-3 个重点讲：

### 5.1 模型选择：Flash vs Pro 双模型分工

> "我选择了 DeepSeek V4，它的 Flash 和 Pro 两个模式各有所长。Flash 响应快(<2s)、成本低，适合逐文件批量扫描；Pro 带 reasoning 推理链、准确率高，适合生成总结和复杂漏洞分析。我把两者组合成一个流水线，用最低成本拿到最高准确率。"

### 5.2 上下文获取：三层架构

> "上下文获取我设计了三层：PR 元信息（标题/描述/commits）→ 文件变更（diff + 全文）→ 项目级（package.json 中的技术栈、tsconfig 配置、import 依赖链）。关联文件通过轻量正则解析 import 语句获取，用正则而不是 AST 是因为 PR 中的代码可能处于编译不通过的状态。"

### 5.3 无服务器 CI 模式

> "传统的 AI review 工具需要部署一个后端服务常驻运行。我这个项目支持 GitHub Actions 模式——PR 触发后直接在 CI runner 上运行分析脚本，分析完通过 GitHub API 回贴评论。整个过程不需要自己的服务器、不需要数据库，零运维成本。"

### 5.4 架构解耦

> "整个系统的每一层都是可替换的。LLM 用的是 OpenAI 兼容的 SDK，想换成别的模型只改一行配置。Embedding 也是独立的，百炼换 OpenAI 也只需要写一个新的 embedding 模块。GitHub API 封装了一层 Service，底层 octokit 换别的 SDK 也不影响上层业务逻辑。"

## 六、未来规划
